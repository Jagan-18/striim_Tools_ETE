# Chapter 1: What is Security?

Security means protecting:

* Data
* Users
* Applications
* Infrastructure
* Networks

from:

* Unauthorized access
* Data theft
* Data modification
* Malware
* Insider threats
* Accidental mistakes

Simple definition:

> **Security ensures that only authorized users and systems can access the right resources in the right way.**

---

# Chapter 2: Why Security Matters

Imagine a bank database.

```text
Customers

Accounts

Credit Cards

Transactions
```

If anyone on the internet could access it:

* Customer data could be stolen.
* Money could be transferred fraudulently.
* Regulations could be violated.

Security prevents these situations.

# Chapter 4: IAM (Identity and Access Management)

This is the **heart of Cloud Spanner security**.

IAM decides:

* Who can access Cloud Spanner?
* What can they do?
* Which resources can they access?

Think of an office building.

People enter using ID cards.

| Person         | Access         |
| -------------- | -------------- |
| Security Guard | Main gate      |
| Employee       | Office floor   |
| HR             | HR room        |
| Finance        | Finance room   |
| Visitor        | Reception only |

Everyone has different permissions.

IAM works the same way.

---

## IAM Components

IAM consists of three parts:

```text
Identity

↓

Role

↓

Resource
```

### Identity

Who?

Examples:

* User
* Group
* Service Account

Example:

```text
devops@company.com
```

---

### Role

What can they do?

Example:

* Read
* Write
* Admin

---

### Resource

Where?

Example:

```text
Cloud Spanner Instance
```

```text
Jagadeesh

```

---

# Chapter 6: Authorization

Authorization answers:

> **What are you allowed to do?**

Example:

Jagadeesh logs in.

Authentication succeeds.

Now IAM checks:

Can Jagadeesh delete the production database?

If not:

```text
Permission Denied
```

Authentication and authorization are different.

# Chapter 7: Service Accounts

This is one of the most important topics for DevOps.

Applications should not use personal accounts.

Instead:

```text
Application

↓

Service Account

↓

Cloud Spanner
```

A service account is a non-human identity used by applications and automation.

Example:

```text
app-prod@project.iam.gserviceaccount.com
```

---

# Why Service Accounts?

Imagine Striim connects to Cloud Spanner.

Should it log in using:

```text
Jagadeesh@gmail.com
```

No.

Instead:

```text
Striim Service Account
```

Benefits:

* Secure
* Auditable
* Easy to rotate credentials
* Not tied to one employee

# Chapter 8: IAM Roles

Cloud Spanner provides predefined IAM roles.

Examples include:

| Role                         | Purpose                      |
| ---------------------------- | ---------------------------- |
| Cloud Spanner Admin          | Full administrative control  |
| Cloud Spanner Database Admin | Manage databases and schema  |
| Cloud Spanner Database User  | Read and write data          |
| Cloud Spanner Viewer         | Read-only access to metadata |

**Example Scenario**

Developer:

Needs:

```text
Read

Write
```

Should they get Admin?

No.

Give only:

```text
Cloud Spanner Database User
```

This follows the **Principle of Least Privilege**.

---

# Chapter 9: Principle of Least Privilege

Very important interview topic.

Rule:

> Give only the permissions required to perform the job.

Wrong:

```text
Everyone = Admin
```

Correct:

```text
Developer

↓

Database User
```

Less risk.

# Chapter 18: DevOps Engineer Responsibilities (Very Important)

This is where you'll spend a lot of your time.

## 1. IAM Management

* Create IAM policies.
* Assign appropriate roles.
* Remove unnecessary access.
* Review permissions regularly.

---

## 2. Service Account Management

* Create service accounts.
* Rotate credentials where applicable.
* Avoid personal accounts in automation.
* Disable unused accounts.

---

## 3. CI/CD Security

Pipeline example:

```text
Developer

↓

Git

↓

Harness

↓

Deploy

↓

Cloud Spanner
```

Security tasks:

* Use service accounts.
* Retrieve secrets securely.
* Avoid hard-coded credentials.
* Limit deployment permissions.

---

## 4. Terraform

Example:

```hcl
resource "google_project_iam_member" "spanner_user" {
  project = var.project_id
  role    = "roles/spanner.databaseUser"
  member  = "serviceAccount:app-prod@project.iam.gserviceaccount.com"
}
```

Infrastructure as Code ensures permissions are version-controlled and repeatable.

---

## 5. Production Access

Production access should require approvals.

Example:

```text
Developer

↓

Approval

↓

Temporary Access

↓

Production
```

Avoid permanent admin access.

---

## 6. Monitoring Security

Watch for:

* IAM changes
* Login failures
* Unexpected service account usage
* Permission denied errors
* Suspicious API activity

---

## 7. Incident Response

Suppose a service account key is leaked.

Actions:

1. Disable the compromised key.
2. Create a new key (or better, use keyless authentication where possible).
3. Update applications.
4. Review audit logs.
5. Verify no unauthorized activity occurred.

# Interview Questions & Answers

### Q1. What is IAM?

**Answer:**
IAM (Identity and Access Management) controls who can access Google Cloud resources, what actions they can perform, and on which resources.

---

### Q2. What is the difference between Authentication and Authorization?

| Authentication                     | Authorization                               |
| ---------------------------------- | ------------------------------------------- |
| Verifies identity ("Who are you?") | Determines permissions ("What can you do?") |

---

### Q3. Why are Service Accounts used?

**Answer:**
Service accounts provide secure, non-human identities for applications, automation, and services to access Google Cloud resources without using personal user accounts.

---

### Q4. Why should DevOps engineers use the Principle of Least Privilege?

**Answer:**
It reduces security risks by granting only the permissions required for a task, minimizing the impact of mistakes or compromised credentials.

---

### Q5. What are the main DevOps responsibilities for Cloud Spanner security?

**Answer:**

* Manage IAM roles and policies
* Create and manage service accounts
* Secure CI/CD pipelines
* Manage secrets
* Monitor audit logs
* Investigate permission issues
* Support incident response
* Automate security with Terraform
